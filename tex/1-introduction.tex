\section{Introduction}

\subsection{Motivation}
In embedded systems development, the contribution of software engineering to overall cost of a product has grown steadily compared to hardware design in recent years~\cite{niespodziany2025}. This trend is driven by continuous advancements in semiconductor technology and the growth in customer demand for broader functionality. Embedded products are released as a family of variants rather than single devices~\cite{clements2001spl}. Since building and maintaining a separate codebase for each variant does not easily scale, the application is decomposed into reusable modules. These modules are often developed and maintained by independent engineering teams. Thus, the industry requires a mechanism to assemble them into a running application without recompiling the entire codebase for each specific product variant.

The \textit{diff} framework~\cite{niespodziany2025} resolves this problem through data-driven composition. In this approach, modules are compiled into a shared binary and register themselves with the framework. At process startup, the framework reads a JavaScript Object Notation (JSON) topology file that explicitly determines which modules to instantiate, how each of them is configured, and the dependency relationships between them. Each topology file represents a completely different application variant from the exact same compiled binary.

However, writing the topology file remains a manual, error-prone process. The developer must ensure its structural correctness while working with limited tool support. This problem enlarges along with the size of the application and the broader ecosystem, such as the number of available modules, variant count, and team size.

This thesis extends the future work identified in Niespodziany's research~\cite{niespodziany2025} by introducing \textbf{\textit{diff-forge}}: a visual editor that allows developers to manage the application's dependencies directly as a graph, while continuing to produce the exact JSON topology file consumed by the existing C++ build pipeline. The next section details the specific gaps that \textit{diff-forge} aims to resolve.

\subsection{Problem Statement}
The manual maintenance of \textit{diff} topology files leaves three distinct gaps in the current workflow. 

\paragraph{Lack of module discovery.}
Available modules are distributed across Artifactory repositories, often with many modules per repository and many versions per module. To compose a new system, the developer must be aware of what modules exist, what interfaces they expose, and what parameters they accept as module configuration.

\paragraph{Lack of graph readability.}
The dependency graph is currently encoded as text within a JSON file. Every architectural inquiry, such as tracing direct or transitive dependencies between modules, requires the developer to manually analyze from the JSON source. Without a visual representation, analyzing the system's structure becomes a time-consuming process.

\paragraph{Lack of validation.}
The JSON parser used during development provides syntactic correctness while lacking the capability of detecting per-entry lexical errors (e.g., invalid identifiers, incorrect references, or malformed values) or graph-structural errors (e.g., cycles, unmet required interfaces, or interface mismatches). These errors are caught only when the binary starts, in continuous integration (CI), or during code review. A broken build blocks other developers, and the responsibility for fixing it often falls to someone other than the original author~\cite{humble2010continuous}.

\subsection{Project Objectives}
The primary objective of this project is to close the three gaps identified in the problem statement while maintaining compatibility with the existing workflow by fulfilling the following criteria:

\begin{itemize}
    \item \textbf{Discover modules from the catalog:} The tool must fetch module metadata at application startup from Artifactory and transform the responses into a searchable catalog panel. This panel should display each module's name, version, and repository source.
    \item \textbf{Compose the graph visually:} The tool must provide a drag-and-drop canvas where modules are represented as nodes and dependencies are represented as directed edges. Users instantiate modules by dragging them from the catalog, connect them by drawing edges, and configure them via inline fields on the nodes. This ensures the dependency graph is both read and edited structurally.
    \item \textbf{Validate and report:} The tool must provide immediate, contextual validation. Lexical errors in each entry are flagged on the node or edge. Graph-structural errors are reported during export with feedback pinned to the offending entries.
    \item \textbf{Export and import JSON topologies:} The tool must allow users to export the graph state to JSON files that comply with existing build pipelines. Additionally, it must allow the user to import a topology file, with validation performed beforehand to report corruption or schema mismatches at the location of the offending entry.
\end{itemize}

\subsection{Thesis Structure}
The remaining chapters of this thesis are structured as follows. Chapter~2 establishes the technical background, covering dependency injection, the \textit{diff} framework, and related work in visual composition. Chapter~3 analyzes the problem in detail, identifying personas, use cases, and architectural drivers. Chapter~4 justifies the architectural design, technology choices, and system layering, along with key design problems. Chapter~5 provides a walk-through of the implementation details.
