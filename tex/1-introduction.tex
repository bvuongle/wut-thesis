\section{Introduction}

\subsection{Motivation}
In embedded systems development, the contribution of software development to the overall engineering cost grows relative to the cost of hardware design~\cite{niespodziany2025}. As semiconductor technology advances and customer demand for broader functionality grows, an embedded application is assembled by dozens of engineers across several teams~\cite{pendharkar2009}, each one producing independent modules that recombine into many product variants over the product family's lifetime~\cite{clements2001spl}.

One response to this assembly problem is to formalize it. The modules are separated from the application that consumes them, and their composition is described in a structure independent of any single module's source code. The \textit{diff} framework, presented by Niespodziany~\cite{niespodziany2025}, applies that idea to C++ embedded software through data-driven dependency injection: the application's composition is described in a JavaScript Object Notation (JSON) topology file consumed at startup.

While \textit{diff} improves reusability and flexibility, composing a product variant means editing the topology file manually. The developer must ensure its structural correctness while working with limited tool support. Humble and Farley describe reliable, repeatable releases as a baseline requirement of modern software engineering~\cite{humble2010continuous}. Manual editing of a topology file does not meet that bar, and the gap grows with the size of the application.

Together, the industry's assembly problem and the manual authoring workflow of the \textit{diff} framework~\cite{niespodziany2025} motivate \textbf{\textit{diff-forge}}~: a visual tool that supports managing the dependency graph and producing a JSON file consumed by the existing C++ build pipeline without modification. This thesis extends \textit{diff}'s data-driven composition with the missing authoring side. The next section sets out the specific gaps that \textit{diff-forge} aims to resolve.

\subsection{Problem Statement}
Manual maintenance of a \textit{diff} topology file leaves three gaps in the developer's workflow. Each grows with the size of the application and the number of developers who work with the file.

\paragraph{Lack of module discovery.}
Modules are distributed across Artifactory repositories, with many modules per repository and many versions per module. To compose a new system, the developer must learn which modules exist, what interfaces they expose, and what parameters they accept. This gap scales with the size of the catalog and the number of new systems assembled.

\paragraph{Lack of graph readability.}
The dependency graph is encoded as text. To trace which components depend on a given module, or whether one component depends on another directly or transitively, the developer reads the JSON or runs a custom query against it. Every architectural question, from the impact of a change to the presence of a cycle, requires this reconstruction.

\paragraph{Lack of validation.}
The JSON parser only checks syntax and is not capable of detecting lexical errors within each entry's configuration fields (such as identifiers, dependency references, or parameter values), or structural errors across the JSON file (cycles, unmet dependencies, interface mismatches). These errors are caught only when the binary starts, in Continuous Integration (CI), or during code review. A broken build blocks other developers, and responsibility for fixing it often falls to someone other than the original author~\cite{humble2010continuous}.

\subsection{Project Objectives}
With the problem defined, the success criteria of this project follow. The tool must close each of the three gaps from the previous section, and must do so without disrupting the existing build flow.

\begin{itemize}
    \item \textbf{Discover modules from the catalog.} At startup, the tool fetches the metadata file published with each module version from Artifactory and assembles the responses into a searchable catalog panel. Each entry displays the module's name, version, and Artifactory source.
    \item \textbf{Compose the graph visually.} A drag-and-drop canvas displays module instances as nodes and their dependencies as edges. The developer instantiates a module by dragging it from the catalog onto the canvas, wires dependencies by drawing edges between nodes, and edits each instance's configuration through inline fields on the node itself. The result is that the dependency graph is read and edited as a graph, not as JSON text.
    \item \textbf{Validate and report.} The editor validates parameter inputs, wired dependency types, and identifier references against the catalog as the developer makes changes. Per-entry errors (invalid parameter value or duplicate instance identifier) are flagged on the offending node or edge. Graph-wide errors (cycles, unmet required interfaces) surface when the user attempts to export the topology file, with feedback on every offending entry.
    \item \textbf{Import and export JSON topologies.} The editor imports an existing topology file so the developer can resume previous work, reports corruption or schema mismatches with the location of the offending entry, and exports the canvas state to JSON files that comply with existing build pipelines.
\end{itemize}

\subsection{Thesis Structure}
The remaining chapters proceed from foundations to evaluation. Chapter~2 covers the technical background: dependency injection, the \textit{diff} framework, and related visual-composition tools. Chapter~3 analyses the problem in detail and derives the architecture drivers. Chapter~4 describes the architectural design of \textit{diff-forge}, including technology choices and codebase layering.
